# Live Interval Cache Scalability Report

**Date:** 2026-05-10  
**Context:** Post-analysis and fixes after a live n=5 main4 run that produced ~131 GB of artifacts.

---

## 1. Problem: 131 GB for a 5-taskset Run

After running `scripts/30_run_yaml_fig4_experiment.py --live` with 5 tasksets and the `main4` algorithm set (SS-tol-fb, UNI-tol-fb, SS-opt, UNI-opt), the repo grew to:

```
artifacts/chunk_cache/   25 GB   (interval cache: ONNX + engines + timing)
artifacts/onnx/          53 GB   (per-variant ONNX copies — REDUNDANT)
artifacts/engines/       53 GB   (per-variant engine copies — REDUNDANT)
```

Total: ~131 GB.  Cache-only rerun still reported `skipped_cache_misses=15389`.

Root cause analysis: see `results/live_cache_artifact_diagnosis.md`.

## 2. New Artifact Layout

### Before (two-level with copies)

```
artifacts/
  chunk_cache/<model>/int_<start>_<end>/
    chunk.onnx              ← source
    chunk_fp32.engine       ← source
    timing.json             ← export_wall_s + build_wall_s only

  onnx/<model>/<variant>/
    chunk0.onnx             ← copy of interval cache
    chunk1.onnx             ← copy (duplicate!)

  engines/<model>/<variant>/
    chunk0_fp32.engine      ← copy of interval cache
    chunk1_fp32.engine      ← copy (duplicate!)
```

Every unique mask created its own variant dir with copies of all ONNX and
engine files, even when those intervals had been profiled before.

### After (single-level, interval cache = canonical)

```
artifacts/
  chunk_cache/<model>/int_<start>_<end>/
    chunk.onnx              ← canonical (built once, shared across masks)
    chunk_fp32.engine       ← canonical
    chunk_fp16.engine       ← canonical
    timing.json             ← export_wall_s + build_wall_s + gpu_mean_ms_fp32
                               + gpu_p99_ms_fp32 (written after profiling)
```

`artifacts/onnx/` and `artifacts/engines/` are no longer created for new runs.

**Storage reduction:** The old layout used O(unique_masks × chunks_per_mask)
storage.  The new layout uses O(unique_intervals) storage.  For n=5 main4 with
~100 masks each sharing 7-interval chunks, this is ~100× reduction.

## 3. What is Mask-Level vs Interval-Level

| Artifact | Granularity | Location | Shared across masks? |
|----------|-------------|----------|----------------------|
| ONNX file | interval | `chunk_cache/<model>/int_<s>_<e>/chunk.onnx` | Yes |
| TRT engine | interval | `chunk_cache/<model>/int_<s>_<e>/chunk_fp32.engine` | Yes |
| GPU timing | interval | `chunk_cache/<model>/int_<s>_<e>/timing.json` | Yes |
| Eval result | mask | `results/evaluations/<model>/<variant>_<prec>.json` | No |
| Profiling cache | model | `results/optimization/.profiling_cache.json` | n/a |

The split config JSON for each mask (generated by `make_selected_split_config`)
now references the interval cache paths directly.  The C++ `table4_runner` reads
engine paths from the config and uses interval cache files with no copies.

## 4. How Cache-Only Mode Now Works

### Before

Cache-only mode only served masks that had an exact eval JSON in
`results/evaluations/`.  Any other mask was counted as `skipped_cache_misses`.

### After

**Two-tier cache-only assembly:**

1. **Mask-level hit**: eval JSON exists → served directly (fast).
2. **Interval timing assembly**: eval JSON missing, but all interval
   `timing.json` files have `gpu_mean_ms_<precision>` and
   `gpu_p99_ms_<precision>` → assemble `EvaluationResult` from per-interval
   timings, write eval JSON, serve as cache hit (no re-profiling).
3. **Skip**: eval JSON missing + some interval timing files lack GPU data
   → counted as `skipped_cache_misses` / `unique_skipped_masks`.

GPU timing is written to `timing.json` after every live profile
(`_backfill_interval_gpu_timing()`).  A one-time backfill utility
(`backfill_interval_gpu_timing_from_evals()`) can populate existing intervals
from already-saved eval JSONs.

### New ProfilingStats counters

| Counter | Meaning |
|---------|---------|
| `skipped_cache_misses` | Attempt count — same mask may appear >1× |
| `unique_skipped_masks` | Distinct masks that were skipped (set-based) |
| `unique_masks_evaluated` | Distinct masks that returned a result |
| `unique_mask_cache_hits` | Distinct masks served from any cache |
| `interval_timing_cache_hits` | Masks assembled from interval timing |

## 5. Cleanup Command

To reclaim the ~106 GB of old variant-specific copies from previous runs:

```bash
# See what would be removed (safe)
python scripts/22_clean_generated_artifacts.py --onnx --engines --dry-run

# Actually remove
python scripts/22_clean_generated_artifacts.py --onnx --engines --yes
```

**Never deleted by this script:**
- `artifacts/chunk_cache/` (interval cache — canonical storage)
- `results/evaluations/` (mask-level eval JSONs)
- `results/table4/` (C++ profiling output)
- `results/optimization/` (profiling cache + algorithm outputs)

## 6. Disk Guard

Scripts 30 and 40 now accept `--min-free-gb N`.  When `--live` is active, the
free space on the repo filesystem is checked before the `LiveProfileBudget` is
created.  If free space < N GB, the run aborts with a cleanup instruction:

```
RuntimeError: Disk guard: only 15.0 GB free on /mnt/storage/tensorrt/...
Run:  python scripts/22_clean_generated_artifacts.py --onnx --engines --yes
```

Recommended threshold: `--min-free-gb 30` on the Jetson storage partition
(currently 15 GB free / 234 GB total after the 131 GB live run).
